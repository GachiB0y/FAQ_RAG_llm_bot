# Security Gateway — дизайн (демо-бот + фундамент прода)

> **Статус:** утверждён 2026-07-14.
> **Контекст:** линия E (Telegram-демо) в [PROJECT_STATUS.md](../../../PROJECT_STATUS.md).
> **Источник требований:** статья В. Викулина «Архитектура надёжных AI-агентов» —
> защитный слой **Gateway (Безопасность)**, которого в проекте нет вообще.

## Зачем

Телеграм-бот (E1) — первый **user-facing** контур проекта (в отличие от внутренних
eval-скриптов). Статья требует, чтобы всё взаимодействие шло через **единый шлюз**
с премодерацией каждого действия. Цель демо — наглядно показать, что «дыра» в
безопасности закрыта. Дизайн сознательно минимальный, но ложится в основу прод-версии.

## Объём

Два контроля (выбраны как самые наглядные для демо и реальные для прода):

1. **Rate-limit по юзеру** — защита бюджета OpenRouter и от DoS.
2. **Prompt-injection guard** — фильтр входящих сообщений до попадания в промпт.

Явно **вне объёма** (YAGNI для демо, кандидаты в прод): access-control allowlist,
PII-маскирование (152-ФЗ пока не в требованиях — храним только `user_id`),
доп. output-guard (порог confidence уже есть в `RAGEngine`), кэш ответов,
авто-фолбэк модели, выделенная guard-модель, лимит по токенам.

## Размещение

Новый модуль `backend/app/core/gateway/` с классом `SecurityGateway`, подключённый
как **FastAPI-dependency** на chat-эндпоинте (`/api/v1/chat`). Проверки отрабатывают
**до** `RAGEngine.query`. Тот же шлюз автоматически защищает и веб-фронт.

```
telegram → bot → POST /api/v1/chat {user_id, text}
   → [SecurityGateway]  1) rate-limit  2) injection-guard
   → RAGEngine.query → {answer, sources}
```

Почему до `query()`: каждый вызов `query()` = платный LLM-запрос в облако + текст
уходит наружу. Шлюз — дешёвый локальный предохранитель перед дорогой/внешней
операцией. Плохой запрос до `RAGEngine.query` не доходит.

## Компонент 1 — Rate-limit по юзеру

- Redis (уже подключён, паттерн как в `backend/app/core/session.py`).
- Ключ `ratelimit:{user_id}:{YYYY-MM-DD}`, счётчик `INCR` + `EXPIRE` (TTL 24 ч).
- **Лимит демо: 10 запросов/день** на `user_id`.
- Превышение → HTTP 429; бот отвечает: «Дневной лимит запросов исчерпан,
  попробуйте завтра».
- Прод-расширение (не сейчас): лимит по токенам вместо числа запросов.

## Компонент 2 — Injection-guard (правила + LLM)

Двухступенчатый, чтобы LLM-вызов был редким и копеечным:

- **Stage 1 — правила:** компилированные паттерны RU+EN (напр. «ignore previous»,
  «забудь инструкции», «system prompt», «ты теперь…», «act as / DAN», попытки
  вытащить системный промпт). Результат: `clean` / `block` / `unsure`.
- **Stage 2 — LLM (только на `unsure`):** классификация через уже имеющийся дешёвый
  `google/gemini-3.1-flash-lite` (OpenRouter) промптом-классификатором →
  `injection: yes/no`. На `clean`/`block` LLM не зовётся.
- Детект (`block` из правил или `yes` из LLM) → запрос до RAG не доходит; бот
  отвечает безопасным отказом: «Не могу обработать этот запрос».

## Видимость для демо

- Каждое решение шлюза структурно логируется: `{user_id, decision, reason, trace_id}`.
- Мини-эндпоинт `/api/v1/gateway/stats` со счётчиками в Redis (`blocked_injections`,
  `rate_limited`) — показать на презентации «сколько атак отбито».
- Полный трейс в Langfuse — когда доедет A3; шлюз пишет тот же `trace_id` для связи.

## Обработка ошибок

| Ситуация | Поведение |
|---|---|
| Превышен лимит | HTTP 429, дружелюбный текст, LLM не зовётся |
| Детект инъекции | HTTP 4xx, безопасный отказ, RAG не зовётся |
| Redis недоступен | fail-open по rate-limit (пропускаем, но логируем WARN) — демо не должно падать |
| LLM-классификатор упал/таймаут | fail-open по Stage 2 (пропускаем как `clean`), логируем WARN |

## Тестирование (TDD)

- **Unit:**
  - rate-limiter на `fakeredis`: 10 проходят, 11-й → 429; сброс после TTL.
  - injection-правила: таблица атак → `block`, нормальные вопросы ФПСР → `clean`.
  - LLM-классификатор замокан: `unsure` → вызывается, `clean`/`block` → нет.
- **Integration:** эндпоинт возвращает 429 при превышении и отказ при инъекции;
  чистый запрос доходит до (замоканного) `RAGEngine.query`.

## Интерфейсы (эскиз)

```python
class GatewayDecision:
    allowed: bool
    reason: str | None          # "rate_limited" | "injection" | None
    trace_id: str

class SecurityGateway:
    async def check(self, user_id: str, text: str) -> GatewayDecision: ...
```

Эндпоинт `/api/v1/chat` вызывает `gateway.check(...)`; при `allowed=False` возвращает
соответствующий статус, иначе идёт в `RAGEngine.query`.
