# uv — вводная (что, как, зачем нам) перед реализацией

**Дата:** 2026-07-06
**Зачем:** понять инструмент ДО кода. Что такое uv, как работает, как настроим под
проект, с примерами. Это Шаг A из MLOps-плана (лечит боль #5 «бардак в окружении»).

---

## 1. Что это в одном предложении

**uv — это быстрый менеджер Python-зависимостей и окружений** (замена pip + venv +
pip-tools + poetry в одном), написан на Rust. Ставит пакеты в 10-100 раз быстрее pip
и **фиксирует точные версии в lock-файле** — чтобы у всех окружение было идентичным.

Аналогия: pip — это «поставь что-нибудь похожее», uv с lock — «поставь ровно эти
версии, байт в байт, как у меня».

---

## 2. Зачем он НАМ конкретно

Вспомни нашу боль:
- `ragas 0.4` сломал `langchain` → мы **вручную** подбирали пины (`ragas==0.2.15`,
  `langchain<0.4`...)
- Зависимости ставили `pip install` прямо в работающий контейнер → при пересборке
  всё терялось, доустанавливали руками (`fastembed`, `tiktoken`, `matplotlib`,
  `rapidfuzz`, `pymupdf`...)
- «Работает на моём маке» — не воспроизводится у другого

| Наша боль | Как uv лечит |
|---|---|
| Конфликты ragas/langchain | uv резолвит совместимые версии и **пишет их в lock** |
| Ручные `pip install` в контейнер | всё в `pyproject.toml`, ставится одной командой |
| Потеря доустановок при пересборке | lock-файл → пересборка даёт то же самое |
| «У меня работает» | `uv sync` даёт байт-в-байт одинаковое окружение |

---

## 3. Как устроено (модель uv)

Два файла — вся суть:

| Файл | Что | Аналогия |
|---|---|---|
| **`pyproject.toml`** | что хотим (зависимости, группы) — пишет человек | «список покупок» |
| **`uv.lock`** | точные версии всего дерева — генерит uv | «чек с точными ценами» |

```
pyproject.toml  (хочу ragas, mlflow, fastapi...)
      ↓  uv lock   (резолвит совместимые версии)
uv.lock         (ragas==0.2.15, langchain==0.3.27, ... — точно и навсегда)
      ↓  uv sync   (ставит ровно из lock)
.venv/          (идентичное окружение у всех)
```

---

## 4. Главные команды (всё что нужно знать)

```bash
uv init                          # создать проект (pyproject.toml)
uv add ragas mlflow fastapi      # добавить зависимости (обновляет lock)
uv add --group eval ragas tiktoken fastembed   # добавить в группу "eval"
uv sync                          # поставить ровно из lock (воспроизводимо)
uv run python scripts/eval.py    # запустить в окружении проекта
uv lock                          # пересчитать lock из pyproject
```

Ключевое отличие от pip: `uv add` не просто ставит, а **записывает в lock**. И
`uv sync` ставит **ровно из lock**, а не «свежайшее совместимое».

---

## 5. Как мы хотим настроить (план под проект)

### Разделяем зависимости на группы
У нас два разных набора: прод-бэкенд и eval-эксперименты. Их надо развести.

```toml
# pyproject.toml (примерный вид)
[project]
name = "faq-rag-bot"
dependencies = [           # прод — то что нужно самому боту
    "fastapi", "llama-index", "qdrant-client",
    "sqlalchemy", "asyncpg", ...
]

[dependency-groups]
eval = [                   # только для экспериментов, не в проде
    "ragas==0.2.15",
    "langchain<0.4",
    "mlflow",
    "fastembed",
    "tiktoken",
    "matplotlib",
    "rapidfuzz",
    "pymupdf",
]
```

Зачем группы: прод-контейнер ставит только `dependencies` (лёгкий), а для eval —
`uv sync --group eval` (тяжёлый, с ragas и прочим). Больше не пихаем eval-пакеты
в прод-образ руками.

### В Dockerfile
```dockerfile
# было: pip install ... + ручные доустановки
# станет:
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen              # ставит ровно из lock, без сюрпризов
```

`--frozen` = «не меняй lock, ставь как записано» — гарантия воспроизводимости.

---

## 6. Что вынесем на реализации

1. Установить uv
2. Перенести текущие зависимости в `pyproject.toml` (+ группа `eval` с нашими пинами)
3. Сгенерировать `uv.lock` (зафиксировать всё что мы подбирали руками)
4. Обновить Dockerfile на `uv sync`
5. Проверить: чистая пересборка → всё работает **без единого ручного** `pip install`

**Готово когда:** `docker compose build backend` даёт рабочее окружение со всеми
eval-зависимостями из lock, без ручной доустановки.

---

## 7. Чего НЕ делаем

- Не переходим на uv для frontend (там pnpm — свой мир)
- Не удаляем pip полностью — uv совместим, можно постепенно
- Не гонимся за экзотикой (uv workspaces и пр.) — базовый sync достаточно

## uv vs что было

| | Было (pip) | Станет (uv) |
|---|---|---|
| Скорость | медленно | 10-100× быстрее |
| Версии | «свежайшее совместимое» (плывёт) | зафиксированы в lock |
| Воспроизводимость | «у меня работало» | байт-в-байт у всех |
| Наши eval-пины | руками в контейнер | в pyproject группе eval |
