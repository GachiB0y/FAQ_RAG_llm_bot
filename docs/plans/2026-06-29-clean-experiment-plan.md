# Чистый эксперимент Dense vs Hybrid — план

**Дата:** 2026-06-29
**Цель:** методологически корректный повторный прогон, исправляющий 3 главные проблемы первого опыта.

---

## Что меняется относительно первого эксперимента

| # | Что было | Что станет | Почему |
|---|---|---|---|
| 1 | Картинка-PDF парсилась в кракозябры | **OCR через Vision-LLM** → реальный текст | Чтобы BM25 цеплялся за осмысленные токены, не мусор. Без этого hybrid получал «лёгкую победу» |
| 2 | `gemma-4-31b` была KG + testset + judge (3 роли одной модели) | **Judge переключается на другую семью** (Meta `llama-3.3-70b` или OpenAI `gpt-oss-120b`) | Чистое разделение «писатель эталонов» и «оценщик» — нет self-bias |
| 3 | KG был построен на 3 PDF, но testset покрывал только «Положение» | **KG + testset перегенерируются полностью** на чистом корпусе из 3 PDF | Чтобы тестсет покрывал ВСЕ документы, не только один |

## Что **остаётся** прежним

- Корпус: те же 3 PDF (Положение о членстве, Правила вида спорта, Картинка-PDF)
- RAG generator: `nemotron-3-super-120b:free`
- KG/testset builder: `gemma-4-31b-it:free`
- Embedding: `bge-m3` (Ollama, локально)
- Sparse: `Qdrant/bm25` (FastEmbed, локально)
- chunk_size=512, top_k=5
- Distribution синтезаторов: 60/20/20
- 3 persona (novice/instructor/lawyer)
- testset_size: 20 (можем поднять до 30 чтобы покрыть больше документов)

## Распределение ролей (финальное)

| Роль | Модель | Семья |
|---|---|---|
| OCR картинки | Vision-LLM (Llama 3.2 90B Vision или Gemini Flash) | вижн |
| KG + Testset builder | `gemma-4-31b-it:free` | Google |
| RAG generator | `nemotron-3-super-120b-a12b:free` | NVIDIA |
| **Judge** | **`meta-llama/llama-3.3-70b-instruct:free`** (fallback: `openai/gpt-oss-120b:free`) | **Meta** (новая, независимая) |
| Embeddings | `bge-m3` (Ollama) | — |

Все 4 LLM-роли — **разные семьи**.

## Этапы

### Этап 0 — Подготовка инфраструктуры (~5 мин)

- [ ] Поднять контейнеры: `qdrant`, `ollama`, `backend` (если не работают)
- [ ] Проверить наличие `bge-m3`, моделей в Ollama
- [ ] Проверить что `kg.json`, `testset_auto.json`, `samples_*.json` — удалены (сейчас старые)
- [ ] Удалить старые коллекции `documents` и `documents_hybrid` из Qdrant
- [ ] Очистить таблицу `documents` в Postgres
- [ ] Проверить наличие моделей-кандидатов на OpenRouter (проба запросов)

**Acceptance:** все сервисы зелёные, доступные модели подтверждены, старые артефакты удалены.

### Этап 1 — OCR для картинки-PDF (~3 мин)

- [ ] Установить `pymupdf` в backend контейнер (~30 сек)
- [ ] Скрипт `backend/scripts/ocr_image_pdf.py`:
  - Берёт `/tmp/corpus/кратко-о-процессе-вступления.pdf`
  - Через `PyMuPDF` рендерит каждую страницу в PNG (in-memory)
  - Для каждой страницы → vision-LLM («Извлеки весь видимый текст на русском, верни verbatim»)
  - Собирает результат → сохраняет как `/tmp/corpus/кратко-о-процессе-вступления.txt`
- [ ] Запуск + ручная проверка результата (читаемый ли русский текст?)

**Acceptance:** TXT-файл с осмысленным русским текстом по содержанию картинки. Если получили мусор — fallback на Tesseract (этап 1b).

### Этап 1b — Fallback: Tesseract (только если 1 не сработал)

- [ ] Добавить `tesseract-ocr`, `tesseract-ocr-rus`, `poppler-utils` через `apt-get` в работающий backend контейнер
- [ ] Установить `pytesseract` и `pdf2image` через pip
- [ ] Тот же скрипт, но через `pytesseract.image_to_string(image, lang='rus')`

**Acceptance:** аналогично — читаемый русский текст.

### Этап 2 — Подготовка корпуса (~2 мин)

- [ ] В скрипте `ingest_local.py` исходный список файлов меняем:
  - `Положение о членстве ОСОО ФПСР.pdf` (без изменений)
  - `Правила вида спорта 2025.pdf` (без изменений)
  - **`кратко-о-процессе-вступления.txt`** (новый файл с OCR-текстом, вместо битого PDF)
- [ ] Запустить `ingest_local.py` → коллекция `documents` (dense only)
- [ ] Запустить `ingest_hybrid.py` → коллекция `documents_hybrid` (dense + sparse)

**Acceptance:**
- В Qdrant 2 коллекции с одинаковым числом точек (~31 + 385 + N чанков из OCR)
- В Postgres 3 записи в `documents` с status='ready'

### Этап 3 — Knowledge Graph (~25 мин)

- [ ] Запустить `generate_kg.py` (KG_PROVIDER=openrouter, gemma-4-31b)
- [ ] Acceptance:
  - Узлов: ~418 (если без `MAX_CHUNKS_PER_DOC`)
  - Рёбер: > 1000 (на большем корпусе должно быть много связей)
  - Топ-entities включают: ФПСР, БЕКОСО, МКПС, статьи Правил, региональные отделения
  - Размер kg.json ~ 5-10 МБ

> Может быть слишком долго на 418 узлах через OpenRouter free (rate-limit + 5 трансформов на узел = ~2000 LLM-вызовов). При этом дневной лимит на free — 2000. Реальный риск: **упрёмся в дневной лимит на этапе KG**.
>
> **Митигация:** ограничиваем «Правила» через `MAX_CHUNKS_PER_DOC=80` → 31 + 80 + N = ~120 узлов. Полностью корпус не покроем, но это все равно лучше первого опыта (там был KG на тех же 31 + 2 узлах).

### Этап 4 — Testset generation (~5-10 мин)

- [ ] Запустить `generate_testset.py` с `testset_size=30` (чуть больше чем в первом опыте, чтобы покрыть 3 документа)
- [ ] Acceptance:
  - 30 пар (вопрос, эталон) в `testset_auto.json`
  - Distribution: ~18 single-hop, ~6 multi-hop abstract, ~6 multi-hop specific
  - Проверить глазами 5-7 вопросов — есть ли по «Правилам» и «Положению» и «картинке-OCR»
  - Если все 30 вопросов про одно «Положение» — есть проблема с KG, надо пересобрать

### Этап 5 — Dense baseline run (~25 мин)

- [ ] Удалить старый кэш `samples_json_dense.json`
- [ ] Запустить `eval_rag.py`:
  - `DATASET_SOURCE=json` (наш testset)
  - `HYBRID=false`
  - `GENERATOR_PROVIDER=openrouter`, `OPENROUTER_GEN_MODEL=nemotron-3-super-120b`
  - `JUDGE_PROVIDER=openrouter`, `OPENROUTER_MODEL=meta-llama/llama-3.3-70b-instruct:free`
- [ ] Acceptance:
  - 30 ответов сгенерированы (RAG-фаза)
  - 120 Job-ов Ragas (30 × 4 метрики) — большинство успешны (NaN < 10%)
  - 4 средние метрики залогированы в MLflow

### Этап 6 — Hybrid run (~25 мин)

- [ ] Удалить кэш `samples_json_hybrid.json`
- [ ] Запустить тот же `eval_rag.py` с `HYBRID=true`
- [ ] Те же acceptance критерии

### Этап 7 — Сравнение и отчёт (~10 мин)

- [ ] Сравнить run-ы в MLflow UI (Compare)
- [ ] Обновить `compare_dense_vs_hybrid.xlsx` через python-скрипт
- [ ] **Записать выводы:**
  - Что hybrid даёт на каждой метрике
  - Сильнее или слабее эффект чем в первом опыте? Почему?
  - Срезы по synthesizer и source_doc — где hybrid побеждает сильнее
- [ ] Опционально: дописать новый отчёт `2026-06-29-clean-experiment-report.md` или обновить старый

## Главные риски и митигации

| Риск | Митигация |
|---|---|
| OCR через Vision-LLM даёт мусор | Fallback на Tesseract (этап 1b) |
| Не хватит дневного лимита OpenRouter на KG (418 узлов слишком много) | `MAX_CHUNKS_PER_DOC=80` → ~120 узлов |
| `llama-3.3-70b` опять 429 как судья | Fallback на `openai/gpt-oss-120b:free` |
| Vision-LLM плохо распознаёт нестандартный шрифт | Принять как факт — оставить только 2 хороших PDF (Положение + Правила), без картинки |
| Hybrid не показывает прирост на чистом корпусе | Это валидный результат — записать как «hybrid не нужен когда retriever получает чистый сигнал» |

## Что _не_ делаем в этой итерации

- Не интегрируем hybrid в прод-код (`backend/app/core/rag/retriever.py`) — это **отдельный шаг 6** из roadmap, после подтверждения результата
- Не делаем MLflow-сервис в docker-compose — sidecar контейнер работает
- Не добавляем CI — после стабилизации
- Не качаем полный корпус с Яндекс-Диска — после Prefect-сетапа

## Принятые решения

1. **OCR подход**: Vision-LLM (PyMuPDF + Llama 3.2 90B Vision или Gemini Flash), fallback Tesseract
2. **Лимит чанков на «Правила»** для KG: `MAX_CHUNKS_PER_DOC=80`
3. **testset_size**: **15** (по 5 на каждый из 3 синтезаторов: single-hop spec / multi-hop abs / multi-hop spec)
4. **Judge**: `meta-llama/llama-3.3-70b-instruct:free`, fallback `openai/gpt-oss-120b:free`

**Ограничение Ragas:** per-document распределение напрямую не задаётся — синтезаторы свободно выбирают узлы из всего KG. После генерации проверяем глазами; если перекос — можно перезапустить с другим random seed (но Ragas detereministичен на одних входных, так что seed контролировать сложно — проще принять что есть).
