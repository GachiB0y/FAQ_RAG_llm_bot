# Флоу работы с моделями: что локально, что через OpenRouter

**Дата:** 2026-07-08
**Зачем:** зафиксировать какие LLM крутим локально (Ollama), какие через OpenRouter,
и **почему** — чтобы решения были обоснованы, а не «наугад».

---

## 0. Выбранный набор моделей (обновлён 2026-07-10) — платный cost/quality

Ушли с free-tier (429, нестабильно) на дешёвые платные — прод-качество за копейки.
Всё через OpenRouter (один баланс, доступ из РФ, **ключ Anthropic не нужен**).
Реальные цены $/1M in/out (июль 2026).

| Роль | Модель | Семья | Цена |
|---|---|---|---|
| **RAG-генератор** (отвечает юзерам, в прод) | `qwen/qwen3.6` ¹ | Alibaba (кит.) | $0.32 / $1.28 |
| **Судья** (eval, независимая семья) | `openai/gpt-5.4` (полная, не mini!) | OpenAI | $2.50 / $15 |
| **KG + testset** (подготовка экзамена) | `google/gemini-3.1-flash-lite` | Google | $0.25 / $1.50 |
| **Эмбеддинги** | bge-m3 | локально | free |
| **OCR** | Tesseract | локально | free |

3 разные семьи (Alibaba / OpenAI / Google) → судья независим от генератора и тестсета
(нет self-preference bias — судья не «подсуживает своим»).
**Цена полного цикла eval ~$0.4–1.5** (~$0.43 на 10 вопросах, ~$1.3 на 30). Судья =
~95% стоимости; генератор и KG — копейки. **Единый источник имён — `backend/models.env`**
(`GEN_MODEL`/`JUDGE_MODEL`/`KG_MODEL`); Makefile его `include`-ит, скрипты берут через env
и своих дефолтов не имеют (нет env → падают). Прод-openai-путь — `Settings.RAG_GENERATOR_MODEL`.

> ¹ Генератор `qwen/qwen3.6` — хороший русский (201 язык), точный вариант (`-plus`/размер)
> уточнить на openrouter.ai. Дешёвая альтернатива `deepseek/deepseek-v4-flash` (~$0.14/$0.28),
> но русский слабее → **прогнать 10–15 вопросов ФПСР глазами до фиксации**; fallback —
> `google/gemini-3.1-flash`. Судья намеренно ПОЛНАЯ GPT (не mini): он определяет доверие
> к метрикам, гоняется только на eval (объём мал → цена не бьёт). Урок 30.06 про self-bias.

> Почему не Claude-судья: подписка Claude Max — это claude.ai + Claude Code, НЕ API;
> в eval-пайплайн (Ragas) её штатно не вставить. А раз всё равно через OpenRouter —
> берём независимую от генератора семью (OpenAI), ключ Anthropic не нужен.

> Роли: генератор ОТВЕЧАЕТ (продукт, в прод); KG/testset ГОТОВЯТ экзамен; судья
> ПРОВЕРЯЕТ. В прод едет только генератор — остальное «леса» для оценки.

## 1. Наше железо (определяет всё)

| Параметр | Значение | Следствие |
|---|---|---|
| Хост | Apple M1, 16 GB RAM | норм для dev |
| Docker лимит | ~10 GB RAM | сюда должна влезть модель + сервисы |
| **GPU в Docker** | **НЕТ** — Docker на Mac не пробрасывает M1 GPU | Ollama в контейнере = **только CPU → медленно** |

**Главный вывод:** наш Ollama крутится в Docker → на CPU. Поэтому:
- модели >7B либо не влезают (OOM), либо мучительно медленные
- судья на локальном qwen2.5:7b был ~8 мин/вызов → eval ~10 часов (мы это прожили)

> ⚡ Нюанс: если бы Ollama работал **нативно на Mac** (не в Docker), он бы
> использовал M1 GPU через Metal → в разы быстрее. Но у нас всё в docker-compose
> ради воспроизводимости. Trade-off: консистентность контейнеров vs скорость GPU.

## 2. Что крутим ЛОКАЛЬНО (Ollama, CPU, ≤10 GB)

| Модель | Размер | Роль | Почему локально |
|---|---|---|---|
| **bge-m3** | 1.2 GB | эмбеддинги | быстрая, приватная, нужна на каждый запрос — не гонять в API |
| **qwen3:1.7b** | 1.4 GB | лёгкий генератор (dev) | быстро прототипировать бесплатно |
| **qwen2.5:7b** | 4.7 GB | генератор/судья (крайний случай) | работает, но медленно — только когда нет альтернатив |

**Правило:** локально — только эмбеддинги (всегда) и мелкие модели для dev.
Тяжёлое и требующее качества — НЕ локально.

**Что НЕ влезло локально (проверено болью):**
- llama3.2-vision:11b → **OOM** (нужно 10.9 GB)
- qwen2.5vl:3b → **OOM** (compute graph 6.7 GB)
- → для vision/OCR ушли на **Tesseract** (системный, не LLM) — оказалось лучше

## 3. Что крутим через OPENROUTER (внешние API)

| Роль | Модель (наш набор) | Почему не локально |
|---|---|---|
| **Судья** (LLM-as-judge) | `openai/gpt-5.4` | нужен сильный + честный; локальный 7B = шум + 10 часов |
| **Генератор** (когда важно качество) | `qwen/qwen3.6` | качество/русский; локально на M1/CPU медленно |
| **KG builder** | `google/gemini-3.1-flash-lite` | сотни вызовов → нужна скорость |
| **Testset generator** | `google/gemini-3.1-flash-lite` | то же |

**Правило:** OpenRouter — всё что требует большой модели ИЛИ скорости ИЛИ качества
оценки. Судья ОБЯЗАТЕЛЬНО не локальный (иначе метрики нечестные/медленные).

**Нюанс OpenRouter free-tier (прожили):**
- лимит ~16-20 запросов/мин + 2000/день на free-модели
- провайдеры периодически 429-ят
- → отсюда боль с ручными рестартами → лечится Prefect (retry) или платным tier

## 5. Как это будет в разных сценариях

### Сейчас (dev/эксперименты)
```
эмбеддинги    → Ollama bge-m3 (локально, всегда)
RAG-генератор → Ollama qwen3:1.7b (быстро/бесплатно, dev) ИЛИ OpenRouter qwen/qwen3.6 (качество)
судья         → OpenRouter openai/gpt-5.4 (никогда локально)
KG/testset    → OpenRouter google/gemini-3.1-flash-lite
OCR           → Tesseract локально
```

### Prod для заказчика (зависит от приватности)
```
Если документы чувствительные (152-ФЗ):
  → всё self-hosted: bge-m3 + LLM через vLLM на GPU-сервере (аренда ~30-100k₽/мес)
  → данные не уходят никуда

Если не чувствительные:
  → эмбеддинги локально + генерация через managed API (YandexGPT/GigaChat для РФ,
    или OpenAI/Anthropic если можно)
  → дешевле, без своего GPU
```

## 5.1. Переносимость генератора OpenRouter → локальный контур (важно!)

**Да, генератор можно позже вынести в локальный контур без переписывания.** Это
и есть смысл абстракции в нашем коде.

Почему легко:
- Наш `eval_rag.py` и `RAGEngine` дёргают LLM через **OpenAI-совместимый API**
  (`base_url` + `model`). OpenRouter — просто один из провайдеров этого API.
- **vLLM / Ollama тоже дают OpenAI-совместимый endpoint.** Значит смена провайдера =
  поменять `base_url` и `model`, **код не трогаем**.

```
Сейчас (dev):   base_url = openrouter.ai/api/v1   model = qwen/qwen3.6
Прод локально:  base_url = http://vllm:8000/v1     model = qwen2.5-72b (на своём GPU)
```

Что нужно для локального прод-контура (когда данные чувствительные, 152-ФЗ):
1. **Железо**: GPU-сервер (аренда ~30-100k₽/мес, напр. с A100/L40).
2. **vLLM** поднимает выбранную модель как OpenAI-совместимый сервис.
3. В конфиге меняем `base_url` на свой vLLM, `model` на локальную (напр. qwen2.5-72b).
4. Данные **никуда не уходят** — весь inference на своём железе.

Важный нюанс: eval мы делаем на `qwen/qwen3.6` (через OR), а прод может быть на
локальной модели → **надо перепрогнать eval на прод-модели**, т.к. качество отличается.
Но пайплайн оценки (Ragas + testset) тот же — просто меняем генератор и смотрим метрики.

**Вывод:** сейчас через OpenRouter — быстро и дёшево для разработки. Перенос в
локальный контур = смена 2 строк конфига + GPU-сервер. Архитектура к этому готова.

## 6. Где тут Langfuse

Langfuse **провайдер-агностичен** — трейсит любые LLM-вызовы, хоть Ollama, хоть
OpenRouter. Поэтому в мониторинге будет видно оба типа моделей единообразно:
latency локального qwen3 vs latency qwen/qwen3.6 через OpenRouter — в одном UI.
Плюс токены и **фактическая стоимость** каждого вызова (см. A3 в PROJECT_STATUS).

## 7. Быстрая шпаргалка «куда какую модель»

```
Нужны эмбеддинги?           → bge-m3 локально. Всегда.
Прототип/дёшево/приватно?   → qwen3:1.7b локально (мелкий, CPU ок)
Нужно качество генерации?   → OpenRouter qwen/qwen3.6 (или потом vLLM)
Судья для eval?             → OpenRouter openai/gpt-5.4. НИКОГДА локально (шум + медленно)
Много вызовов (KG/testset)? → OpenRouter google/gemini-3.1-flash-lite (скорость)
OCR/сканы?                  → Tesseract (не LLM, локально, бесплатно)
Vision-LLM?                 → OpenRouter (локально OOM на M1/Docker)
```

**Одним принципом:** локально — дёшево/приватно/мелко (эмбеддинги + dev-генератор);
OpenRouter — качество/скорость/размер (судья, big-генератор, KG/testset).
