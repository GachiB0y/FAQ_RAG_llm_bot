# Langfuse — вводная (что, как, зачем нам) перед реализацией

**Дата:** 2026-07-08
**Зачем:** понять инструмент ДО кода. Это Шаг A3 из MLOps-плана — лечит боль #4
«нет мониторинга», самую зияющую дыру проекта.

---

## 1. Что это в одном предложении

**Langfuse — это observability для LLM-приложений:** он записывает КАЖДЫЙ запрос
к твоему боту со всеми шагами внутри (retrieval → промпт → ответ LLM), меряет
latency, стоимость, токены, и показывает это в UI + позволяет оценивать качество
на живом трафике.

Аналогия: если MLflow — это «лаборатория» (эксперименты до прода), то Langfuse —
это «чёрный ящик самолёта» + «приборная панель» для живой системы в проде.

## 1.1. Бесплатный или платный? (как с Prefect — не путаться)

| Вариант | Что | Цена |
|---|---|---|
| **Langfuse (OSS)** | self-hosted через docker-compose | **бесплатно** (MIT/Apache) |
| **Langfuse Cloud** | их хостинг + командные фичи | платно (есть free-tier) |

**Мы используем OSS self-hosted** — поднимаем локально в Docker, ноль затрат.
Страница цен на сайте — про Cloud, нам не нужен.

## 2. Зачем он НАМ конкретно

Сейчас у нас **ноль видимости** в то, что происходит с RAG в работе:
- Пользователь задал вопрос → мы не знаем сколько шёл ответ, сколько стоил,
  какие чанки нашлись, не соврал ли бот
- Если качество деградирует — узнаем только когда пожалуются

| Наша боль (#4) | Как Langfuse лечит |
|---|---|
| Не видим что происходит с запросами | трейс каждого: retrieval + generation по шагам |
| Не знаем latency/стоимость | считает время и $/запрос автоматически |
| Не ловим галлюцинации в проде | online-eval (Ragas-метрики на семпле трафика) |
| Нет данных для улучшения | 👍/👎 + плохие ответы → новые тест-кейсы (Фаза 8) |

## 3. Как устроено (модель данных Langfuse)

```
Trace (один запрос пользователя = «Что такое БЕКОСО?»)
 ├── Span: retrieval           (поиск в Qdrant, 120ms)
 │     input: вопрос
 │     output: 5 чанков
 ├── Generation: llm-answer    (вызов LLM, 2.3s, $0.004, 340 токенов)
 │     input: промпт + контекст
 │     output: ответ
 │     model: gpt-oss-120b
 └── Score: faithfulness=0.9   (оценка, опц. — online eval или 👍/👎)
```

| Понятие | Что |
|---|---|
| **Trace** | один запрос пользователя от начала до конца |
| **Span** | шаг внутри (retrieval, обработка) — время |
| **Generation** | вызов LLM — токены, стоимость, модель, промпт+ответ |
| **Score** | оценка трейса (метрика, 👍/👎, online-eval) |
| **Session** | цепочка трейсов одного диалога (чат-история) |

## 4. Как это подключается к коду (главное)

Два способа обернуть наш RAG:

### Способ 1 — декоратор `@observe` (просто)
```python
from langfuse import observe

@observe()                       # ← Langfuse сам создаёт trace
def rag_query(question: str):
    chunks = retriever.search(question)      # станет span
    answer = llm.generate(question, chunks)  # станет generation
    return answer
```

### Способ 2 — интеграция с LlamaIndex (автоматом)
Langfuse умеет хукать LlamaIndex — тогда retrieval + LLM трейсятся **без ручных
декораторов**, как `mlflow.llama_index.autolog()` но для observability.

```python
from langfuse.llama_index import LlamaIndexInstrumentor
LlamaIndexInstrumentor().start()   # всё, дальше все query_engine трейсятся сами
```

## 5. Как мы хотим настроить (план под проект)

### Поднять сервис
Langfuse — это несколько контейнеров (web + worker + postgres + clickhouse для
аналитики). У них есть готовый `docker-compose.yml`.

```bash
# отдельный compose, чтобы не мешать нашему основному стеку
git clone https://github.com/langfuse/langfuse
cd langfuse && docker compose up -d          # UI на localhost:3000
```

### Подключить наш RAG
1. Создать проект в Langfuse UI → получить `LANGFUSE_PUBLIC_KEY` + `SECRET_KEY`
2. Ключи в `.env.eval` (рядом с OpenRouter, тоже локально)
3. Обернуть `RAGEngine.query()` через `@observe` или LlamaIndex-инструментор
4. Сделать 5-10 запросов через чат → увидеть трейсы

### Что хотим увидеть
- Граф каждого запроса: retrieval (сколько чанков, время) → LLM (токены, стоимость)
- Latency-разбивку: где тормозит — поиск или генерация
- Стоимость запроса в деньгах (Langfuse знает цены моделей)

## 7. Langfuse vs MLflow (не дублируют)

| | MLflow (есть) | Langfuse (внедряем) |
|---|---|---|
| Фаза | 3-4 разработка (офлайн) | 7 прод (онлайн) |
| Единица | прогон = весь eval-набор | трейс = один запрос юзера |
| Стоимость запроса | не считает | считает $/запрос |
| Фокус | «какая конфигурация лучше» | «что сейчас с живой системой» |

Аналогия: MLflow ≈ тесты перед релизом, Langfuse ≈ Sentry/Grafana в проде.
Нужны оба, на разных этапах.

---

## 8. Чего НЕ делаем

- Не Langfuse Cloud — только self-hosted OSS
- Не тащим clickhouse-аналитику на максимум — базовый трейсинг сначала
- Не заменяем MLflow — Langfuse дополняет (прод-слой), не конкурент

## Итог: что попробуем

Поднять Langfuse локально → обернуть RAG → увидеть **живые трейсы с latency и
стоимостью**. Это первое «окно» в то, что происходит с системой в работе — то,
чего у нас сейчас нет вообще.
