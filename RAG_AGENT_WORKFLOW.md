# 🔄 RAG Agent Development Workflow — как мы строим RAG-агентов

> Референс-карта жизненного цикла RAG-агента: от «пришёл заказчик» до «работает
> в проде и улучшается». Отвечает на вопрос **«какой инструмент, на каком шаге,
> зачем»**. Обращаемся к этому документу когда нужно понять «а где мы сейчас» и
> «что дальше по-хорошему».
>
> Актуальный статус проекта — в [PROJECT_STATUS.md](PROJECT_STATUS.md).

---

## Главный принцип: Evaluation-Driven Development

Сначала строим **систему оценки**, потом под неё разрабатываем — как TDD, только
для AI. Оптимизировать RAG без метрик = гадать вслепую.

> Мы это доказали на себе: в первом эксперименте hybrid «улучшил» метрики на +51%,
> но при честной методологии (независимый судья + чистый корпус) выигрыш исчез.
> Оптимизировали на артефактах измерения. См.
> [clean-experiment-report](docs/plans/2026-06-30-clean-experiment-report.md).

Три мысли, которые определяют весь флоу:
1. **Eval-first** — система оценки раньше оптимизации.
2. **Данные > модель** — 50% качества решается на парсинге/чанкинге, а не выбором LLM.
3. **Петля обратной связи** — прод не финиш, а источник данных для улучшения.

## Фазы жизненного цикла

Легенда статуса: ✅ пройдено · 🔵 идём сейчас · ⬜ впереди

### Фаза 0 — Discovery / постановка задачи ✅

Превращаем «хочу бота по документам» в измеримое ТЗ.

| Определяем | Зачем |
|---|---|
| Какие документы, сколько, формат | Парсинг, OCR, объём |
| Кто пользователи, какие вопросы | Personas + тестовый набор |
| Что значит «хороший ответ» | Метрики успеха (без них проект не защитить) |
| Приватность (152-ФЗ) | Облако vs self-hosted |
| Нагрузка (запросов/день) | Затраты + инфраструктура |

**Итог фазы:** golden-вопросы от заказчика (10-20 реальных пар вопрос/ответ) — фундамент оценки.
**Инструменты:** пока никакие технические — разговор + документ.

---

### Фаза 1 — Данные (самое недооценённое) ✅

Документы → чистый текст → чанки → вектора. **Здесь решается 50% качества.**

| Шаг | Инструмент | Зачем |
|---|---|---|
| Парсинг | `pypdf`, `python-docx`, `unstructured` | Текст из документов; `unstructured` понимает таблицы/заголовки |
| OCR | **Tesseract**, vision-LLM | Когда документ = картинка/скан |
| Чанкинг | LlamaIndex splitters | Разбить на куски — тут прячется много качества |
| Эмбеддинги | **bge-m3**, Qwen3-embedding | Текст → вектора (1024-dim) |
| Векторка | **Qdrant** / pgvector | Хранить и искать вектора |
| Версии данных | **DVC** | «Эти чанки = этот результат» воспроизводимо |

**Наш опыт:** битый картинка-PDF давал кракозябры и портил метрики, пока не добавили Tesseract OCR.

---

### Фаза 2 — Прототип RAG (baseline) ✅

Первый работающий «вопрос → ответ». **Не оптимизируем — цель просто чтобы работало.**

```
вопрос → embedding → поиск в Qdrant (top-k) →
    [чанки + вопрос] в промпт → LLM → ответ + источники
```

| Компонент | Инструмент | Зачем |
|---|---|---|
| Orchestration | **LlamaIndex** | Склеивает retriever + LLM + промпт |
| LLM (dev) | **Ollama** локально | Быстро и бесплатно прототипировать |
| Retrieval | Qdrant dense | Найти релевантные чанки |

---

### Фаза 3 — Оценка (сердце подхода) ⭐ ✅

Строим систему, меряющую качество числами.

| Шаг | Инструмент | Зачем |
|---|---|---|
| Тестовый набор | **Ragas TestsetGenerator** + golden заказчика | Вопросы + эталоны + personas |
| Knowledge Graph | Ragas | Для multi-hop вопросов |
| Метрики | **Ragas** — faithfulness, context precision/recall, answer relevancy | Числовая оценка |
| LLM-судья | **независимая** сильная модель (не та что генерит!) | Оценивает без self-bias |
| Трекинг | **MLflow** | Сравнивать прогоны в UI |

**Метрики что значат:**
- **Faithfulness** — нет ли галлюцинаций (ответ опирается на контекст)
- **Answer Relevancy** — отвечает ли на вопрос
- **Context Precision** — не приволок ли retriever мусор
- **Context Recall** — нашёл ли retriever всё нужное

---

### Фаза 4 — Итерации ✅

Крутим параметры, каждый раз меряем, сравниваем в MLflow. Меняем **одну вещь за раз**.

Что перебираем: chunk_size · top_k · dense vs hybrid · embedding-модель · промпт ·
LLM-генератор · reranker (bge-reranker/Cohere — часто +качество).

| Инструмент | Зачем |
|---|---|
| **MLflow Compare** | Видеть «параметр X → метрика ↑/↓» |
| **DVC pipelines** | Воспроизводимость коммит → данные+конфиг+результат |

---

### Фаза 5 — MLOps-обвязка 🔵 МЫ ЗДЕСЬ

Делаем разработку воспроизводимой (лечим «7 болей»). Трекер:
[mlops-implementation-status](docs/plans/2026-07-06-mlops-implementation-status.md).

| Боль | Инструмент | Зачем |
|---|---|---|
| Хаос окружения | **uv** + lockfile / Docker | Одинаково у всех |
| Ручной труд | **Makefile** → **Prefect/Dagster** | `make eval`; пайплайны по расписанию |
| Версионирование | **DVC** + MLflow Registry | Данные/промпты/модели под версиями |
| Тестирование | **CI (GitHub Actions)** + Ragas в CI | PR краснеет при просадке метрик |

---

### Фаза 6 — Деплой ⬜

Выкатываем в прод.

| Компонент | Инструмент | Заметка |
|---|---|---|
| API | **FastAPI** ✓ | Точка входа |
| LLM serving | **vLLM** / SGLang / managed API | Ollama НЕ для прода (нет батчинга) |
| Контейнеры | **Docker** ✓ + K8s (если масштаб) | Упаковка |
| CI/CD | GitHub Actions | Автодеплой при merge |

---

### Фаза 7 — Мониторинг в проде ⭐ ⬜

Видим что происходит с живой системой (сейчас у нас этого ноль — боль #4).

| Что | Инструмент | Зачем |
|---|---|---|
| Каждый LLM-запрос | **Langfuse** | Latency, стоимость, трейс шагов |
| Качество в проде | Langfuse online-eval / Ragas на семпле | Не деградировала ли модель |
| Инфра | **Prometheus + Grafana** | CPU/RAM/RPS/ошибки |
| Drift | **Evidently** | Вопросы юзеров уплыли от корпуса? |
| Обратная связь | 👍/👎 в UI | Юзеры сами размечают качество |

---

### Фаза 8 — Непрерывное улучшение ⬜

Замыкаем петлю: прод-данные → улучшение системы.

```
Юзеры задают вопросы  →(Langfuse логирует)
  → находим где RAG плохо ответил (👎, низкий score)
  → добавляем эти вопросы в golden testset
  → прогоняем eval, находим причину (retrieval? промпт?)
  → фиксим → CI проверяет что не сломали остальное
  → деплоим → снова мониторим → (цикл)
```

Flywheel: чем дольше работает, тем умнее становится.

## Как пользоваться этим документом

- **Начинаешь новую RAG-задачу?** Иди по фазам 0→8 сверху вниз.
- **Не понимаешь что за инструмент?** Найди его в таблице фазы — там «зачем».
- **Не знаешь что делать дальше в текущем проекте?** Смотри 🔵-фазу + [PROJECT_STATUS.md](PROJECT_STATUS.md).
- **Обновляй статусы** (✅🔵⬜) по мере продвижения.
